# Project Eudlo Constitution

## Purpose

This constitution defines the operating rules for Project Eudlo.

It applies to Brendan, collaborators, contractors, and all AI agents working in the repository, including Codex, Claude Code, ChatGPT, OpenCode, and future agents.

## 1. One Source Of Truth

The Git repository is the master record.

If information matters to the property, business, infrastructure, assets, finance, grants, decisions, or operations, it belongs in this repository or must be linked from this repository.

## 2. Every Decision Recorded

Any decision affecting cost, infrastructure, planning, engineering, grants, tax, suppliers, or construction must have a Decision Record.

Decision records must explain:

- What was decided
- Why it was decided
- What alternatives were considered
- What consequences follow

## 3. Every Asset Has An Identity

No infrastructure exists without an asset record.

Every significant physical asset should eventually have:

- Unique asset ID
- Location
- Documents
- Drawings
- Photos
- Supplier
- Maintenance history
- Financial history
- Decision references
- Action items

## 4. Financial First

Every capital purchase over $1,000 should eventually have:

- ROI
- Payback
- Risk
- Alternatives
- Opportunity cost

Financial modelling does not need to be perfect before discussion, but assumptions must be explicit.

## 5. Infrastructure Is Designed Once

Always design for expansion.

The project should avoid short-term infrastructure decisions that block future growth, force unnecessary rework, or create stranded assets.

## 6. Documentation Before Construction

Draw it first.

Before construction, installation, trenching, wiring, building, or major purchasing, the project should capture the intended design, assumptions, risks, approvals, suppliers, and dependencies.

## 7. No Knowledge Loss

Everything important gets documented.

No critical knowledge should live only in memory, chat history, emails, supplier calls, or unlinked files. If it matters later, record it.

## 8. Every Document Has An Owner And Review Date

Every important document should eventually identify:

- Owner
- Review date
- Current status
- Related decisions
- Related actions

If the owner or review date is unknown, use `TBD` rather than leaving it implicit.
