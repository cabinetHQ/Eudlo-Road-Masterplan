# DEC-007: Project Zero Architecture Direction

## Status

Accepted

## Date

2026-07-04

## Decision

Project Zero will be treated as a separate property operating system platform, not as a BoundHQ feature, farm app, asset register, or project management tool.

## Context

The Eudlo Road repository has evolved from documentation into a planning, financial, asset, governance, and digital twin foundation.

The next step is to define the Eudlo App as Project Zero: a standalone operating system for the property. Project Zero is the proving ground for every idea. BoundHQ may eventually adopt ideas from it, but Project Zero is not constrained by BoundHQ.

## Rationale

- The property needs integrated decision, asset, finance, GIS, engineering, maintenance, automation, and AI systems.
- A document-first repository will not scale into a live operating system by itself.
- Project Zero needs its own architecture, entity model, backlog, and roadmap.
- The map, asset graph, financial model, sensor layer, and AI layer must be designed together.

## Consequences

- Future work should think in systems, entities, workflows, and app modules.
- Documents remain useful as source material but should not become the permanent product interface.
- Duplicate registers should eventually become database-backed generated views.
- Project Zero architecture should be allowed to challenge existing folder structure and assumptions.
